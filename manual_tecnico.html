1Introducción y objetivos

La Plataforma de Análisis Estadístico Académico es un sistema web diseñado para la Dirección Académica de una institución de educación superior. Su propósito central es automatizar el cálculo de los indicadores institucionales oficiales y presentarlos como dashboards interactivos, eliminando el trabajo manual en hojas de cálculo y reduciendo el riesgo de errores en los reportes.

Usuarios objetivo

  • Administrador del sistema: configura cuentas, carga los CSVs históricos, supervisa procesos.
  • Directivo: consulta dashboards, exporta reportes, revisa la bitácora de auditoría.
  • Usuario analista: visualiza indicadores filtrados por subsistema y período.

2Arquitectura general

El sistema sigue una arquitectura de tres capas clásica, completamente contenerizada con Docker Compose. Todos los servicios se comunican dentro de una red interna privada (internal); solo el servidor web expone el puerto 80 al exterior.

Además de las variables de entorno anteriores, el sistema define valores internos por defecto:

6Instalación y despliegue

Requisitos previos

  • Docker 24 o superior
  • Docker Compose v2 (docker compose sin guion)
  • 4 GB de RAM disponibles (recomendado 8 GB)
  • Puerto 80 libre (producción) o puertos 5173 y 8000 (desarrollo)

Instalación en producción

Un único comando levanta todo el sistema:

./start.sh

El script start.sh realiza automáticamente:

  1. Verifica que Docker esté instalado y en ejecución.
  2. Genera el archivo .env con secretos aleatorios seguros (solo la primera vez).
  3. Construye todas las imágenes Docker.
  4. Inicia los servicios (PostgreSQL, Redis, Backend, Worker, Frontend/Nginx).
  5. Espera a que PostgreSQL esté listo (health check).
  6. Aplica todas las migraciones de Alembic.
  7. Crea el usuario administrador inicial.
  8. Imprime en consola la URL de acceso y las credenciales generadas.
Resultado final La plataforma queda accesible en http://localhost (o el puerto configurado). Las credenciales del administrador se muestran en la terminal al finalizar.

Restricciones de integridad importantes

  • matricula: clave única compuesta por (subsistema_id, ciclo_escolar, cuatrimestre, programa_educativo) — evita duplicados al re-subir el mismo CSV.
  • users → subsistemas: ON DELETE SET NULL — si se elimina un subsistema, el usuario queda sin subsistema asignado pero no se borra.
  • matricula → subsistemas: ON DELETE CASCADE — si se elimina un subsistema, se eliminan todos sus datos.
  • Contraseñas: almacenadas como hash Argon2, nunca en texto plano.

9Seguridad

Contraseñas

Almacenadas con Argon2 (ganador del concurso Password Hashing Competition). Es el algoritmo más resistente a ataques de fuerza bruta por GPU.

Tokens JWT

Firmados con HMAC-SHA256 (HS256). Access token caduca a los 15 minutos; refresh token a los 7 días. La clave de firma se genera aleatoriamente en cada instalación.

Rate Limiting

Máximo 120 peticiones por minuto por IP, aplicado globalmente con SlowAPI. Configurable con RATE_LIMIT_PER_MINUTE.

CORS

Solo los orígenes listados en ALLOWED_ORIGINS pueden hacer peticiones cross-origin. Configuración estricta por lista blanca.

Validación de entrada

Todos los request bodies son validados con Pydantic v2 antes de procesarse. Los archivos CSV pasan por validación adicional de tipos y valores permitidos.

Auditoría

Las acciones relevantes (login, carga de datos, cambios de usuario) quedan registradas en la tabla audit_logs con usuario, timestamp y recurso afectado.

RBAC (Control de acceso basado en roles)

El acceso a cada ruta se controla mediante dependencias inyectables en FastAPI (app/api/deps.py). El JWT incluye el campo role que se verifica en cada petición sin consultar la base de datos (salvo el refresh).

10Carga de archivos CSV / Excel

El sistema acepta archivos en formato CSV (delimitado por comas) o Excel (.xlsx, .xls). Los nombres de columnas son insensibles a mayúsculas y espacios laterales se eliminan automáticamente.

Proceso de carga paso a paso

  1. El administrador selecciona el archivo y el tipo de dataset en la interfaz de carga (Cargas).
  2. El frontend envía el archivo al endpoint POST /api/v1/uploads.
  3. El backend guarda el archivo en disco (/data/uploads/) y encola una tarea en Redis.
  4. Un worker Celery toma la tarea y ejecuta la validación y carga.
  5. El frontend hace polling cada 3 segundos hasta que el estado cambia.
  6. Al terminar, el job reporta filas válidas, filas con error y detalle de errores.

Estados del job de carga

EstadoDescripción
pendingEl job fue creado, espera ser tomado por un worker
processingEl worker está leyendo y validando el archivo
completedCarga exitosa (puede tener filas con error reportadas)
failedError grave (p.ej. columnas faltantes, archivo corrupto)

11Fórmulas e indicadores

Las fórmulas estadísticas están centralizadas en backend/app/services/formulas.py para facilitar su verificación y auditoría.

Matrícula actual

Calcula la matrícula efectiva descontando bajas y sumando nuevo ingreso.

Matrícula Actual = Total − Bajas Reprobación − Bajas Deserción + Nuevo Ingreso

Porcentaje (base de todos los indicadores relativos)

Porcentaje = (Numerador / Denominador) × 100

Si el denominador es 0, el resultado es 0 (sin división por cero).

Tasa de Reprobación

% Reprobación = (Bajas Reprobación / Total Matrícula) × 100

Tasa de Deserción

% Deserción = (Bajas Deserción / Total Matrícula) × 100

Eficiencia Terminal

Se calcula sobre un máximo de 3 generaciones para que el indicador sea comparable entre períodos.

% Eficiencia Terminal = (Egresados / Matrícula Generacional) × 100

Tasa de Titulación

% Titulación = (Titulados / Egresados) × 100

Aprovechamiento Académico

El promedio ponderado se calcula directamente desde el dataset evaluacion_academica:

Promedio General = Σ(promedio_pe × num_pe) / Σ(num_pe)

Evaluación Docente

Se calcula el promedio de puntaje separado por tipo de evaluador:

Promedio Docente = Σ(puntajes) / N evaluaciones [por tipo: alumno | directivo]
Cobertura de edad escolar (indicador opcional) Si el CSV de matrícula incluye poblacion_edad_escolar y egresados_nms, el sistema puede calcular:
% Cobertura = (Total Matrícula / Población Edad Escolar) × 100

Gestión de estado

TanStack Query (datos del servidor)

Gestiona las peticiones a la API, caché automático, revalidación y estados de carga/error. Elimina la necesidad de manejar manualmente los useEffect de fetching.

Zustand (estado del cliente)

Almacena el estado de filtros activos (subsistema, ciclo escolar, cuatrimestre) que se comparten entre todos los dashboards. También guarda el token JWT y los datos del usuario autenticado.

  • Líneas: tendencia temporal por ciclo escolar
  • Pastel / Dona: distribución por género o tipo
  • Barras apiladas: composición de matrícula (altas, bajas, permanencia)

El componente ProtectedRoute verifica el token JWT y el rol antes de renderizar cada página. Si el token expiró, redirige a /login. Si el rol es insuficiente, redirige al dashboard.

13Subsistemas

La plataforma es multi-tenant por subsistema: cada dato (matrícula, evaluaciones, docentes) está asociado a un subsistema mediante subsistema_id. Los usuarios pueden estar asignados a un subsistema específico, lo que limita los datos que pueden ver.

Los subsistemas se pre-cargan en la migración 0002_seed_subsistemas.py:

Aislamiento de datos Un usuario con subsistema_id = 3 solo verá los datos de Interculturales. Un administrador sin subsistema asignado ve todos los datos de todos los subsistemas.

14Roles y permisos

El sistema implementa tres roles con permisos diferenciados. El rol se almacena en la tabla users y se incluye en el payload del JWT para verificación sin base de datos.

Flujo de gestión de usuarios

1. Admin inicia sesión 2. Admin crea nuevo usuario con: nombre, email, contraseña, rol, subsistema 3. Usuario recibe sus credenciales por canal externo 4. Usuario inicia sesión → recibe JWT con su rol 5. En cada petición: backend verifica rol del JWT → permite o rechaza

Gestión de contraseñas

  • Las contraseñas nunca se almacenan en texto plano.
  • Al crear un usuario, el backend hashea la contraseña con Argon2 antes de guardarla.
  • El administrador puede resetear la contraseña de cualquier usuario editando el perfil.
  • No existe actualmente un flujo de "recuperar contraseña" por email (requiere configuración de servidor SMTP externo).

15Exportación de reportes PDF

El backend genera reportes PDF mediante WeasyPrint y templates Jinja2. WeasyPrint convierte HTML+CSS a PDF de alta calidad, permitiendo incluir gráficas y tablas formateadas profesionalmente.

Proceso de generación

  1. El usuario selecciona el tipo de reporte y el período desde la interfaz.
  2. El frontend envía POST /api/v1/reports/pdf con los parámetros de filtro.
  3. El backend consulta los datos en PostgreSQL.
  4. Renderiza el template Jinja2 con los datos.
  5. WeasyPrint convierte el HTML resultante a PDF.
  6. El PDF se devuelve como descarga directa (Content-Type: application/pdf).

Exportación desde el navegador

Adicionalmente, cada dashboard tiene botones de exportación rápida que usan las bibliotecas del frontend:

  • PDF de pantalla: usa html2canvas + jsPDF para capturar la vista actual.
  • Excel: usa xlsx para exportar los datos tabulares directamente.

17Migraciones de base de datos

El esquema de la base de datos se gestiona con Alembic. Las migraciones están en backend/alembic/versions/ y se aplican automáticamente al iniciar con start.sh.

Historial de migraciones

Comandos de Alembic

# Aplicar todas las migraciones pendientes
docker compose exec backend alembic upgrade head

# Ver el estado actual
docker compose exec backend alembic current

# Ver el historial
docker compose exec backend alembic history

# Crear una nueva migración (tras modificar un modelo)
docker compose exec backend alembic revision --autogenerate -m "descripcion_del_cambio"

# Revertir la última migración
docker compose exec backend alembic downgrade -1
Advertencia Nunca ejecutes alembic downgrade en producción sin un respaldo completo de la base de datos. Las migraciones de reversión pueden eliminar datos permanentemente.

18CI/CD

El repositorio incluye un workflow de GitHub Actions en .github/workflows/deploy.yml configurado para ejecutarse en un runner self-hosted (el propio servidor de la institución).

Flujo del pipeline

git push → GitHub │ ▼ GitHub Actions (runner self-hosted en el servidor) │ ├── 1. Checkout del código ├── 2. Construir imágenes Docker ├── 3. Ejecutar pruebas (pytest + vitest) └── 4. Si pasan → docker compose up -d --build → alembic upgrade head → Sistema actualizado sin downtime

Configuración del runner self-hosted

El runner se instala directamente en el servidor de producción. Ventajas:

  • No se expone el servidor a Internet; GitHub Actions envía los jobs al runner.
  • El deploy ocurre sin necesidad de SSH externo ni credenciales adicionales.
  • Las imágenes Docker se construyen localmente en el servidor, sin registry externo.

Programar respaldos automáticos Se recomienda configurar un cron job en el servidor para ejecutar el respaldo diariamente y almacenarlo en un destino externo (NAS, S3, etc.).

Si el volumen de archivos a procesar es alto, se pueden escalar los workers Celery sin reiniciar el resto del sistema:

# Escalar a 4 workers

docker compose up -d --scale worker=2
